## 1. CubeSatの本質的特性

Part 1の「フォーキャスト型」アイデアは誰でも思いつく。ここでは**CubeSatの制約を逆手に取る**発想で再構成する。

### 1.1 制約の反転

| 従来の解釈（ネガティブ） | 反転（ポジティブ） |
|--------------------------|---------------------|
| 短命（**1年**） | **はかなさ＝桜、蛍、花火** |
| 低SLA（届くかわからない） | **不確実性＝おみくじ、運試し** |
| 消える（大気圏再突入） | **流れ星になる＝願いの対象** |
| 1基では弱い | **世代交代＝輪廻、継承** |

日本には「はかなさ」「無常」を愛でる文化がある。これと接続する。

### 1.2 類推（アナロジー）

- **桜**：散るから美しい
- **蛍**：短い命の光
- **花火**：一瞬の輝き
- **線香花火**：消える瞬間が最も美しい
- **灯籠流し**：送って消える
- **流れ星**：願いを込める対象

### 1.3 コスト構造（市場調査）

#### 打ち上げコスト

| 手段 | 価格 | 備考 |
|------|------|------|
| J-SSOD（ISS放出） | $90,000/1U（約1,400万円）※FY22 | 振動条件緩和、クルーチェック可 |
| SpaceX Rideshare | $5,000〜6,500/kg（2025年） | 最低$300,000/50kg〜 |
| Rocket Lab Electron | $7〜8M/打上げ | 専用打上げ |

**1U（約1.3kg）の打ち上げ概算**：
- J-SSOD経由：約1,000〜1,500万円
- SpaceX Rideshare：約100〜200万円（ただしデプロイヤー・インテグレーション費用別途）
- **Falcon 9相乗りの価格低下トレンドあり**（2022年→2025年で変動）

Sources: [SpaceX Rideshare](https://www.spacex.com/rideshare), [SatBase](https://satbase.com/articles/cubesat-launch-costs), [Bright Ascension](https://brightascension.com/how-much-does-it-cost-to-launch-a-cubesat/)

#### 機体コスト（1U〜3U）

| 項目 | 市場調査（参考） | 実績ベース（FM品） | 備考 |
|------|------------------|---------------------|------|
| 構造体（フレーム） | 20〜50万円 | 20〜30万円 | ISIS等の市販品 |
| 太陽電池パネル（SAP） | 20〜60万円/枚 | **40〜50万円/枚** | 複数枚必要、展開式は高価 |
| EPS（電源系） | 50〜100万円 | **100万円〜** | **1U向け縮退傾向、入手困難化** |
| OBC（コンピュータ） | 30〜80万円 | **200万円** | FM品（宇宙実績あり）は高価 |
| 通信系（UHF） | 70〜150万円 | 100〜150万円 | S/X帯は高価 |
| ADCS（姿勢制御） | 50〜150万円 | 100〜200万円 | ミッション次第で省略可 |
| ミッション機器 | 0〜数百万円 | - | 相乗りペイロード次第 |

**実績ベース試算（1U、FM品）**：
- フレーム：20万円
- SAP（2枚）：80〜100万円
- EPS：100万円
- OBC：200万円
- 通信系：100万円
- **合計：500〜600万円**（ADCS省略時）

**注意点**：
- 市場調査の価格は「カタログ価格」「海外安価品」を含む
- **FM品（宇宙実績あり、審査不要）は高価**
- **特に電源系（EPS）は1U向け製品が縮退傾向、入手困難性が上昇中**
- 安価な製品はEM（エンジニアリングモデル）止まり、FM審査に追加コスト発生リスク

Sources: [NanoAvionics](https://nanoavionics.com/blog/how-much-do-cubesats-and-smallsats-cost/), [Bright Ascension](https://brightascension.com/what-is-a-3u-cubesat-and-when-would-you-use-one/), BSP調達実績

#### コスト構造の相互関係（複雑系）

```
ミッション要件
    │
    ├─→ 必要コンポーネント決定 → 機体コスト
    │
    ├─→ 必要電力 → 太陽電池サイズ
    │                   │
    │                   ↓
    │              ペイロード容量への影響
    │                   │
    │                   ↓
    │              相乗り余地（増/減）
    │                   │
    │                   ↓
    │              コスト分担可能性
    │
    ├─→ 通信要件 → UHF/S/X帯選択 → コスト・重量
    │
    └─→ 姿勢制御要件 → ADCS要否 → 省略可能性
```

**箇条書き：コスト最適化の相互依存**

- **ミッション次第で省けるコンポは多い**
  - 姿勢制御（ADCS）：精密な指向不要なら省略可
  - 推進系：軌道維持不要（1年で再突入前提）なら省略可
  - 高度な通信系：UHF store&forwardなら低コスト

- **電力バジェット ↔ ペイロード容量**
  - ミッション機器の消費電力が小さい → 太陽電池を小型化可能
  - 太陽電池が小さい → ペイロードスペースに余裕
  - 余剰スペース → **相乗りペイロードを載せられる**

- **相乗り余地 ↔ コスト分担**
  - 相乗りペイロードが増える → コスト分担者が増える
  - 1事業者あたりコストが下がる → 事業化ハードルが下がる
  - ただし相乗り調整コスト（MI機能）が発生

- **打ち上げ手段選択 ↔ コスト/スケジュール/リスク**
  - J-SSOD：実績あり、条件緩和、ただし高価
  - Falcon 9相乗り：安価だが調整複雑、スケジュール不確実
  - 専用打上げ：高価だが自由度高い

**結論：コスト最小化は「ミッションの絞り込み」と「相乗りの最大化」の同時達成が鍵**

- MI機能が「相乗り可能性」を技術判断 → 1基の衛星に複数ミッションを載せる
- コンソ/BSPが「実装」 → 衛星を作り、運用する

**宙の種がなければ需要がバラバラ → 相乗りが成立しない → コストが下がらない → 事業化できない**

## 4. テレメトリ派生の3方向（根源的アイデア）

### 4.1 根源

**衛星テレメトリ（HKデータ）＋ 電波ノイズ**

- 温度、電圧、姿勢角、太陽電池出力、通信強度...
- ランダム性を持つ ＝ 人間には制御できない
- 「宇宙の意志」「宇宙からの声」として解釈可能

### 4.2 派生の3方向

```
              衛星テレメトリ／電波ノイズ
                       │
        ┌──────────────┼──────────────┐
        ↓              ↓              ↓
    おみくじ       芸術系        対話・メッセージ
   （運勢生成）  （楽曲・画像）   （宇宙からの声）
```

| 派生 | 内容 | 対応アイデア |
|------|------|--------------|
| **おみくじ** | HKデータ→運勢生成（純粋にデータから） | 3.9 宇宙おみくじ |
| **芸術系** | 電波ノイズ→音楽/サウンド、軌道→ビジュアル | 3.3 衛星アート×ビエンナーレ |
| **対話・メッセージ** | 意図×ランダム性（運営の意図を衛星が選ぶ） | 3.1 スローコミュニケーション、3.4 見守り |

### 4.3 「宇宙のメッセージボトル」（意図×ランダム性）

**メッセージの内容・タイミングの両方にランダム性を持たせる**＝海に流したボトルのように、いつ届くか、何が届くかわからない。

- **内容のランダム性**：テレメトリでメッセージを選択（4.2の設計）
- **タイミングのランダム性**：衛星パス・通信状態で届くタイミングが変わる
- **宛先のランダム性**（派生）：誰に届くかも衛星が選ぶ

**「宇宙のメッセージボトル」に包含されるアイデア**：
- 3.1 スローコミュニケーション（タイミングのランダム性）
- 3.4 見守りメッセージ（内容×タイミング）
- 3.9 宇宙おみくじ（内容のランダム性）
- 対話bot全般（意図×ランダム性）

## 5. 補足アイデア（深化の種）

### 5.1 衛星と人生を重ねる（ライフサイクル同期）

**子供が生まれた年に打ち上げ、1歳の誕生日頃に消える。**

- 衛星の寿命（1年）と人生の節目を同期
- 「誕生衛星」「入学衛星」「結婚衛星」
- 衛星が消える時＝人生のフェーズが終わる時
- 次の衛星に「記憶」を継承、新しいフェーズへ
- **七五三、成人式と接続**

### 4.2 衛星ガチャ（ランダム縁結び）

**どの衛星に割り当てられるかは運次第。**

- 登録時にランダムで「担当衛星」が決まる
- 衛星によって「性格」「運勢」が違う設定
- 同じ衛星の人同士で「同衛星コミュニティ」形成
- **推し活文化、星座占いと接続**

### 4.3 消えるから託せる（宇宙焚き火）

**燃やして消したいものを、衛星に載せる。**

- 後悔、秘密、未練、遺言、手紙
- 衛星が消える時、一緒に「燃えて消える」
- 地上では燃やせないもの（データ、デジタル遺品）を宇宙で焚く
- 「消えることが救い」になるサービス

### 7.2 構造的働きかけの仮説（汎用メソッド）

#### フェーズ1：制約の共有（技術理解の最小化）

**目的**：非技術者が「何ができて、何ができないか」を直感的に理解する

| 共有すべき制約 | 伝え方（非技術者向け） |
|----------------|------------------------|
| 衛星は1年で消える | 「1年限りの命。桜や蛍のよう」 |
| 通信は常時ではない | 「1日数回、数分だけ話せる遠距離恋愛」 |
| 確実に届く保証はない | 「届くかわからない手紙。おみくじのよう」 |
| コストは1基1,000万円 | 「5事業で割れば1事業200万円」 |

**メソッド**：技術者が15分で「制約」だけをプレゼン。機能ではなく制約を伝える。

---

#### フェーズ2：反転の促し（制約→価値）

**目的**：制約をネガティブに捉えず、価値に転換する思考を促す

| 促しの問い |
|------------|
| 「1年で消える」を前提に、何を載せたい？ |
| 「届くかわからない」なら、何を送りたい？ |
| 「数分しか話せない」なら、何を伝える？ |

**メソッド**：ワークショップ形式。付箋で書き出し → グルーピング → 派生展開

---

#### フェーズ3：相乗りの概念共有

**目的**：「自分の事業だけでは成立しない」ことを理解し、協調の必然性を認識する

```
「あなたのアイデアは年間○万円の収益見込み」
        ↓
「衛星1基の維持には年間1,000万円必要」
        ↓
「他の○つのアイデアと組み合わせると成立する」
        ↓
「誰と組むか？」
```

**メソッド**：収益試算ワークシート → 相乗りマッチング → コスト分担の合意形成

---

#### フェーズ4：MIへの接続

**目的**：アイデアを「技術仕様」に翻訳可能な形に整理する

| 非技術者が出すもの | MIが翻訳するもの |
|--------------------|------------------|
| 「こういう体験を作りたい」 | 通信要件（頻度、データ量） |
| 「このタイミングで届けたい」 | 軌道・パス計算 |
| 「この人に届けたい」 | ユーザー管理・認証 |

**メソッド**：「要望シート」のテンプレート化 → MI側が技術可否を判定 → フィードバック

### 7.3 場の設計原則（仮説）

| 原則 | 内容 |
|------|------|
| **制約先行** | 機能より制約を先に共有。制約が創造性を生む |
| **反転思考** | ネガティブをポジティブに変換する問いを投げる |
| **相乗り前提** | 単独成立を目指さない。組み合わせを前提に議論 |
| **技術分離** | 技術者は「可否判定」に専念。アイデア出しは非技術者 |
| **収益可視化** | 早い段階で収益試算。現実感を持たせる |

## 8. 事業化フロー（Phase 0）

### 8.1 前提

- 現段階では「MI機能を持つ独立法人」を立てるのは時期尚早
- 宙の種は「場の提供」に徹し、契約当事者にならない
- 契約は「個人/個社 ↔ コンソ」の直接契約
- 実績ができた段階（Phase 1以降）で組織形態を再検討

### 8.2 基本フロー

```
┌─────────────────────────────────────────────────────────┐
│ 1. アイデア成熟（宙の種内）                              │
│    ・複数回のWSで事業案を練る                           │
│    ・収益試算、相乗りマッチング                         │
│    ・要望シート作成                                    │
└─────────────────┬───────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────────────────┐
│ 2. MI協議（技術可否判定）                               │
│    ・MI機能担当者（コンソ内、暫定）が対応               │
│    ・「できる/できない/条件付き」を判定                 │
│    ・概算見積（レンジ）を提示                          │
└─────────────────┬───────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────────────────┐
│ 3. 事業化判断（個人/個社）                              │
│    ・参加者自身が「やる/やらない」を決定               │
│    ・やる場合：自社で資金調達 or 共同出資を募る         │
│    ・宙の種は判断に関与しない（情報提供のみ）          │
└─────────────────┬───────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────────────────┐
│ 4. 契約（個人/個社 ↔ コンソ）                          │
│    ・契約当事者：事業化を決めた個人/個社               │
│    ・相乗りの場合：複数者が共同発注 or 代表者契約      │
│    ・契約テンプレート（コンソ側で用意）を使用          │
│    ・REG確認：規制・許認可の論点整理                   │
└─────────────────┬───────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────────────────┐
│ 5. 実装（コンソ→技組）                                 │
│    ・コンソが受注、技組に展開                          │
│    ・スケジュール・マイルストーン設定                  │
│    ・進捗報告は発注者（個人/個社）へ                   │
└─────────────────────────────────────────────────────────┘
```

### 8.3 利益相反の回避策

| 懸念 | 対策 |
|------|------|
| 宙の種が案件を誘導 | 宙の種は「場の提供」のみ、契約当事者にならない |
| MI判定の恣意性 | MI判定は複数名で実施、議事録を残す |
| コンソ内での需給癒着 | 契約・見積プロセスを文書化、第三者（県等）が監査可能に |

### 8.4 論点リスト（今後詰める）

#### 8.4.1 相乗り時の調整ルール

| 論点 | 選択肢（例） |
|------|-------------|
| 意思決定 | 多数決 / 出資比率 / 幹事社一任 |
| 途中撤退 | 可（返金なし）/ 不可 / 代替者必須 |
| 追加要望 | 全員合意 / 幹事社判断 |

#### 8.4.2 MI協議のコスト

| フェーズ | コスト案（例） |
|----------|---------------|
| 初回相談（要望シートレベル） | 無料 |
| 概算見積（レンジ提示） | 無料 |
| 詳細検討（技術仕様書作成） | 有料（受注時充当） |

#### 8.4.3 最低発注条件

- 最低金額の有無
- MI判定のクリア要件
- 発注者の与信（法人/個人、前払い比率等）

#### 8.4.4 知財の扱い

| 選択肢 | 内容 |
|--------|------|
| A. 発案者帰属 | アイデアは出した人のもの |
| B. 共有 | 宙の種/県も利用権を持つ |
| C. パブリック | 誰でも使える |

※現時点の仮置き：「発案者帰属、ただし宙の種内で共有された情報に秘密保持義務なし」

### 8.5 Phase移行の目安

| Phase | 状態 | 組織形態 |
|-------|------|----------|
| 0（現在） | 実績なし、試行段階 | 宙の種は任意団体、直接契約 |
| 1 | 実績1-2件、パターン化 | 契約テンプレート整備、標準フロー確立 |
| 2 | 継続的需要あり | MI機能の法人化 or 既存法人移管を検討 |
